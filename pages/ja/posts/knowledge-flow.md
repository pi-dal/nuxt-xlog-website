## AIが水流となるとき

もう2026年の夏だ。AIは空気のように、日常のあらゆる隙間に染み込んでいる——メールを書き、情報を調べ、コードを編集し、あなたの話し相手にさえなる。でも、立ち止まってよく見てみると、不思議なことに気づく：私たち個人の知識世界は、まだそこに立ち止まったままのように見える。

読んだ本、行った実験、授業中にひらめいたアイデア、深夜に書き留めた思索……それらはあらゆる場所に散らばっている。ここにノート、あそこに会話、どこかの深夜のドキュメントで眠る霊感。それらの間には橋がなく、まるで孤島のようだ。

AIはすぐそばにあるのに、それらの断片の向こう側にいるあなたを読むことができない。

鍵は「コンテクスト」という三つの字にある。

AIが人を理解できるのは、一回の対話で入力された数行のテキストのせいではない。本当に価値のあるものは、あなたが長い時間をかけて積み上げてきた軌跡だ——何を読んだか、何を繰り返し考えたか、どこで行き詰まったか、どうやってそこから抜け出したか。それらの軌跡の総和が、記憶だ。

記憶のないAIとの対話は、毎回が初対面だ。あなたは何度も同じことを言い、AIは何度も同じことを忘れる。

でも問題は、記憶をどこに置くべきかということだ。

[ChatGPT](https://chat.openai.com) は先週あなたが尋ねたことを覚えている——しかしそれは同社のサービスシステムの中に閉じ込められていて、あなたには見えず触れもしない。[Claude](https://claude.ai) には Project Knowledge があり、長期的なコンテクストを詰め込める——でもそれはプロジェクトのためであって、あなたという人間のためではない。[Notion AI](https://www.notion.so) の Q&A はワークスペースを検索できる——でもそれはチームコラボレーションのロジックであって、個人の知識が成長する方法ではない。[Obsidian](https://obsidian.md) はすべてをローカルのマークダウンファイルに保存する——オープンでコントロール可能だ。しかしその記憶は静的で、AIの参加がなければ、ファイルはただのファイルであり、自分で繋がりを育てることはない。

すべてのプロダクトがメモリを実装している。しかし、すべてのプロダクトのメモリは孤島だ。

[ChatGPT](https://chat.openai.com) で書いた下書きは、自動的に [Obsidian](https://obsidian.md) のノートに現れたりしない。[Claude](https://claude.ai) で行った深い分析は、次の対話に直接引き継がれない。読書ノートは [WeRead](https://weread.qq.com) に、コードの断片は [VS Code](https://code.visualstudio.com) のコメントに、あなたの考えはあなたの頭の中に——それらはすべてあなたのものなのに、ドアのない家に散らばっている。

ドアのない家——それが断片が流動できない真実だ。記録のツールが十分でないからではない——記憶が打ち砕かれて散らばっているからだ。

一つのノート、一つの会話、推导中の詰まり、実験失敗の記録——それらはフォルダの中の沈黙したアーカイブであるべきではないし、あるプロダクトのサーバーに囚われるべきでもない。それらの断片がネットワークに織り上げられ、AIがあなたの歩んだ道筋を辿れるようになったとき、それらはもはや死んだデータではなく——真にあなたのものである川となる。

「収集」から「貫通」へ、「記録」から「流動」へ。これはツールの問題であり、より重要なのは、知識を見る一つの方法なのだ。

## 行き止まりの支流

2025年末、私はある河道を修理しようとした。

読んだすべての論文や技術記事を細かく砕き、整然と巨大な貯水池に積み上げ、AIに言った：必要なときに自分で取りに来い。

実際にやった。文書を分割し、ベクトル化し、インデックスを作り、コンテクストを組み立てた——すべてが整然としていて、精密な配管システムのようだった。

しかし水が実際に流れ始めたとき、水路が間違った場所に接続されていることに気づいた。

最初の問題は、後になってこう理解した：**一つの記事は、決してパラグラフの総和ではない。**

それには呼吸があり、起伏がある。著者が最初のページで仕掛けた伏線は、最後の章でようやく回収される。コンテクストは一本の川であり、あなたはそれを個々の器にすくい上げている。ベクトル検索はその器を見つけられるが、その川がどこに流れているかは知らない。あなたがAについて尋ねると、Aとラベル付けされた器を引き上げる——しかしその記事は実際にはAを反駁していた。川が断ち切られているから、それは知らないのだ。

二つ目の問題が続く：**それはあなたについていけない。**

今日新しい論文を読み、明日新しいアイデアが浮かび、明後日には昨日の結論を覆す。知識は流動しているが、貯水池は静止している。更新のたびに再分割、再マッピング、再インデックスが必要だ。やがて、それを維持する時間が使う時間を上回る。あなたは知識と向き合っているのではなく、配管と向き合っていることに気づく。

三つ目の問題は、最も静かなものだ：

**忘却が消えていた。**

良い記憶システムは、何を覚えるべきかを知っているだけでは不十分だ。何を手放すべきかも知っているべきだ。しかしこの倉庫では、何も手放されない。すべての断片が等しく扱われ、重要なものと取るに足らないものが同じ場所を共有している。日が経つにつれ、すくい上げるものは増えていくが、残す価値のあるものは減っていく。ノイズがシグナルをかき消す。

私はその水路を埋めた。水が通れなかったからではない。その水の流れ方が、川のあるべき姿とはあまりにかけ離れていたからだ。

しかし水はまだ流れる必要があった。

その時期、私は [Claude Code](https://claude.ai/code) でプロジェクトを書いていた。最初はただの便利なツールだと思っていたが、次第に何か違うものに気づいた。

Claude Code の出力は「完璧」ではなかった——時にコードは遠回りし、時に変数名はいい加減で、時に自分が書いたものを自分でリファクタリングした。しかし不思議なことに、プロジェクトは前に進んでいた。しかも速かった。

さらに重要なことに、私は自分がまったく異なる方法でそれを使っていることに気づいた。

私はもはやそのオペレーターではなかった。毎ステップ何をすべきか考えたり、細かい指示に分解したり、理解してもらうための適切な言葉遣いを選んだりする必要がなかった。ぼんやりとした方向性を言うだけだ——「このインターフェースを修正して」「なぜ遅いか調べて」「この機能を実装して」——すると自分でファイルを読み、問題を特定し、コードを書き、デバッグし、自分のバグを修正する。

私が必要としていたのは**協働者**であり、**ツール**ではなかった。

これが後に私が理解する「Agent Native」だ——AIをより良い検索エンジンやテキストジェネレーターとして扱うのではなく、独立して行動できる存在とすること。それは独自のコンテクスト、独自のツール、独自の記憶を持つ。毎ステップ何をすべきか教える必要はなく、あなたの習慣、プロジェクトの構造、コーディングスタイルを知っている。あなたが口を開く前に、必要なものを準備しておくことができる。

私は立ち止まって考えた。

かつてソフトウェアを書くとき、私たちはアーキテクチャの完璧さを追求した。各層の責任は明確に、各モジュールはテスト可能で交換可能に、依存性注入、デザインパターン、クリーンアーキテクチャ——これらすべては、最初に道を舗装しておくことで、後続の人が迷わないようにするためだった。

しかし Claude Code はそうは働かない。アーキテクチャの美しさなど気にしない。それが気にするのは、この思考の線が実行できるかどうか、この機能が数分で結果を出せるかどうかだ。粗く書くが、素早く反復する。回り道を恐れない——AIにとって回り道のコストはほぼゼロだからだ。

私は気づき始めた：

**探求のコストが十分に低いとき、完璧はもはや目標ではない。スピード、変更可能性、破棄可能性——これらが優雅さよりも重要になる。**

アーキテクチャが重要でないということではない。極めて低いコストで繰り返し試行錯誤できるとき、「まず動かしてからリファクタリングする」方が「考えてから行動する」よりも実際の創造プロセスに近いのだ。川が設計図の上で描かれてから改道するのではないのと同じだ——良いソフトウェアはもはや描かれた設計図ではなく、絶えずコースを変える川なのだ。

これは私が埋めた水路と同じ教訓だ。

あの貯水池を造ったとき、私は知識に対してアーキテクチャをするのと同じやり方をしていた——各層を先に設計し、各インターフェースを定義し、すべてを適切な場所に置いた。しかし知識はコードではない。アーキテクチャは必要ない——必要なのは流動だ。最初から完璧である必要はない——あなたについていける必要があるのだ。

ソフトウェアがこのように書けるのなら、なぜ知識も同じようにできないのか？XD

## 雨の後の川

水路を埋めた後、私は周囲を観察し続けた。このことを考えているのは私だけではないことに気づいた。

2025年末から2026年初めにかけて、パーソナルエージェントが雨後の筍のように現れた。それぞれが同じ問いに答えていた：AIはどのようにあなたを記憶すべきか？

yetone の [Alma](https://alma.now) は、私の目に最初に留まったものだ。美しいチャットインターフェースと記憶システムを持っている——あなたの好みやコンテクストを覚え、何を覚えているかを見ることができ、それらの記憶を自分で管理できる。ウェブ検索やツール呼び出しもサポートしている。しかし最初に私を惹きつけたのはこれらの機能ではなく、その背後にある考え方だった。Alma は最初から「パーソナルAIアシスタント」として設計されていた。単なるチャットボットの殻ではなく、独自の記憶、独自のツール、独自のアイデンティティを持っている。覚えているのは一回の対話ではなく、あなたという人間だ。

次に来たのが [OpenClaw](https://docs.openclaw.ai)（当時はみんな「小さなロブスター」と呼んでいた）。Agent Native を開発者コミュニティから一般の視野に初めて持ち込んだものだ。本当に特別だったのはその存在様式だ——IMの中に生きている。チャットアプリで話しかけると、ブラウザ、ターミナル、エディタ、ファイルシステムを操作する。日常のチャットと仕事の環境が、ここで初めて繋がった。

そして [Bub](https://github.com/bubbuild/bub)。Bub は「非常に古風な」アイデアを使った——テープベースの記憶。すべてのコンテクストを線形のタイムラインに保存し、カセットテープのように。いつでも巻き戻して、ある日の会話や、ある決断の瞬間を訪れることができる。その記憶は検索されるのではなく、時間の糸を逆に辿って見つけられる。

そして私自身の **Abraxas**。[GitHub](https://github.com/pi-dal/Abraxas) でオープンソース化されている——対話を通じて成長するパーソナルエージェントだ。チームもプロダクトマネージャーもなく、その成長の仕方自体がこのアイデンティティを反映している：スキル（Markdown操作知識）とプラグイン（ホットロード可能なPythonツール）。コードを変更する必要はなく、Markdownファイルを書くだけで新しいスキルになる。

そのコアはNOUS——対話を通じて自己代謝するコントロールプレーンだ。覚えているのは「あなたが言ったこと」ではなく、「あなたがどんな人間か」だ XD

Alma が澄んだ川なら、OpenClaw は外海に接する河口、Bub はテープレコーダー——そして Abraxas は川底で自ら神経系を進化させた生き物のようだ。記憶とアイデンティティを一つに成長させている。

これらのエージェントはそれぞれ異なる方向性を持っているが、一つだけ共有していることがある：

**記憶はもはや付属品ではなく、核心である。**

各エージェントは独自の方法であなたを記憶する。注意深く作成されたプロンプトによってでも、ベクトルデータベースによってでもなく、あなたとの日々の協働を通じて蓄積されたものによって。その記憶は生きており、成長する——詰め込まれたものではない。

これは私にとって、個人知識システムにおける巨大な飛躍だ。

「私が記録し、AIが検索する」から「AIがあなたと共に歩みながら、自然にあなたを記憶する」へ。

具体的な例を二つ挙げよう。

一つ目はRSSだ。以前は [Inoreader](https://inoreader.com) で購読していた——毎日何百もの記事を読み、忘れ、時々 [Instapaper](https://www.instapaper.com) に保存し、それらの記事は永遠にそこに眠っていた。その後、エージェントにこれをやってもらうようにした：毎日RSSをスキャンし、私の研究に関連する記事を選び、読み、要約を書き、Obsidianのノートに保存し、以前読んだ記事との関連性を注釈する。もう読書を「管理」する必要はない——それは流れの一部になった。

二つ目は [X](https://x.com) だ。多くの研究者や開発者をフォローしており、彼らは興味深い論文のリンクや技術的考察を頻繁に共有している。以前の習慣は：スクロール→いいね→ブックマーク→忘れる。今はエージェントに定期的にそれらのタイムラインをチェックさせ、実質的な投稿を見つけ、リンク先の論文や記事を読み、再び要約をノートに書き、以前記録した関連トピックとタグ付けする。情報摂取が「たまたま目にしたもの」から「フィルタリングし堆積してくれる川がある」へと変わった。

どれも複雑ではない。しかし一つの可能性を示してくれた：

**エージェントがあなたの知識と情報の steward になったとき、あなたはもはや知識自体を管理する必要がない。ただ自分のことをしていればよく、残りの川は自ら流れる。**

## Alma × Nowledge Mem：2月から4月のワークフロー

水路を埋めた後も、河床はまだそこにあった。

2026年2月、私は新しい方向に掘り始めた。今回は貯水池を作りたくなかった——二つの川を合流させたかった。

一つは [**Alma**](https://alma.now)。一つは [**Nowledge Mem**](https://github.com/nowledge-co/nowledge-mem)。

それらはツールではない。あるいは、ツールであるだけではない。それらは水路の二つの形態だ——一方は流動を、もう一方は堆積を担当する。

### Alma：底の見える川

Alma は yetone が作った。使えば使うほど気づく——その記憶パネルは開かれている。これはUIの細部ではない。

設定に隠されていない。二級メニューに埋められていない。開けばすぐに、それが何を覚えているかが見える——ずっと前に言ったこと、あるトピックに対するあなたの態度、調整した好み。すべての記憶がそこにある。編集でき、削除でき、「これは私じゃない」と伝えることができる。

<img src="/assets/knowledge-flow/alma-memory.webp" alt="Alma の記憶パネル。表示・編集可能な短期記憶リストと検索エリア。" width="900">

多くのメモリシステムを使ってきた者には、これがどれほど贅沢なことかわかる。

ほとんどのAIの記憶は目に見えない深い潭だ。その深さは未知で、AIは教えてくれない。正しく覚えていることを期待するしかない。しかしAlmaは違う。覚えているものが見え、覚えていないものもわかる。この川は澄んでいて、すべての小石の位置が見える。

そしてそれらの小石はあなたが投げ入れたものではない。あなたが一緒に働くうちに、水流が自然に河床に運んだものだ。使えば使うほど、河床は豊かになる。一度限りの知識注入ではなく、流れの中での自然な堆積だ。

これはブラックボックス的な記憶とは全く異なる。他の川が底を見せなければ、泥の下に何が眠っているか永遠にわからない。しかしAlmaは透明だ。その記憶はあなたが与えたデータではなく、あなたと一緒に働いた後に自然に沈殿したものだ。

そしてAlmaには一つの水源しかないわけではない——[DeepSeek](https://chat.deepseek.com) は急流で速く浅い、[GPT](https://openai.com) は深水域で沈殿に時間がかかる、[Google](https://gemini.google.com) は複数の細流が集まるデルタ地帯だ。それらはすべて同じ河口から入ってくる——異なるウィンドウを行き来する必要はない。一つの入口、複数の支流。

しかし使い続けて気づいた：Almaの記憶はAlmaの中でしか流れない。

[Claude Code](https://claude.ai/code) や [Codex](https://github.com/openai/codex) で行ったコード分析は、Almaの記憶パネルには見つからない。それらのコンテクストはそれぞれの部屋に閉じ込められている——ドアは鍵がかかっていないが、廊下もない。

これはChatGPTのサーバーに囚われた記憶と同じだ。ただ少し広い部屋に変わっただけ。

一本の川が真に生きるためには、一つの水路だけでは足りない。

### Nowledge Mem：河床そのもの

Alma が流れる川なら、[Nowledge Mem](https://github.com/nowledge-co/nowledge-mem) は河床だ——すべての川が最終的に流れ着き、交わり、堆積し、結晶化する場所。

[Nowledge Mem](https://github.com/nowledge-co/nowledge-mem) は静かに自身を位置づけている：「記憶層」。チャットツールでもノートアプリでもない。ただの基礎だ。[Alma](https://alma.now) の水も、[Claude Code](https://claude.ai/code) の水も、[Codex](https://github.com/openai/codex) の水も、[Gemini CLI](https://cloud.google.com/gemini-cli) の水も、真夜中に突然浮かんだアイデアも——すべての水がここに集まる。

<img src="/assets/knowledge-flow/nowledge-mem-graph.webp" alt="Nowledge Mem のメモリグラフビュー。エンティティ、記憶、アーティファクト、スレッド間の関係ネットワーク。" width="1000">

その構造はシンプルだ。三つの言葉：

**Trace**。水そのものの痕跡。言ったことはそのまま、加工しない。

**Unit**。水から拾い上げた小石——一つの結論、一つの好み。

**Crystal**。小石が非常に長い間転がされた後、透明で安定になり、もはや容易に流されなくなる。

Trace から Unit へ、そして Crystal へ——それは泥が水晶になるプロセスだ。古いものを捨てるのではなく、時間が古いものの上に新しいテクスチャを成長させる。

すべての小石は自分がどこから来たかを覚えている。すべての結晶化は上流の方向を帯びている。今では論文全体を丸ごと放り込むこともできる——自分で読む。そしてこの知識からwikiを成長させる——wiki-linkを使って関連する記憶を繋ぎ、地形図のように、各河床の上流に何が流れていたかを教えてくれる。

<img src="/assets/knowledge-flow/nowledge-mem-wiki.webp" alt="Nowledge Mem のライブラリ/ウィキビュー。知識ベースのドキュメントエントリとその可読コンテンツ。" width="1000">

また、私が貯水池を造ったときにできなかったこともやってのける：何を忘れるべきかを知っている。

毎回のクエリは、六つの異なる方法を並行して実行し、あなたが探しているものを見つける——意味で探し、キーワードで探し、エンティティ関係で探し、知識グラフのノードで探す——そして結果を新鮮さと関連性で混合・ランク付けする。各記憶は減衰スコアを持ち、時間が経つほど軽くなり、頻繁に参照されるものは重くなる。

重要なものは浮かび上がる。時代遅れのものは沈んでいく。もう必要のないものは、河床の奥深くに徐々に消えていく——削除されるのではなく、流れそのものによって優しく押しのけられる。

また、**EVOLVES** というメカニズムもある。各知識と過去の他の記憶との間には、明確な関係線がある——「置き換えた」「豊かにした」「反駁した」「深めた」。あなたの知識は散らばったビーズではなく、時間の中で自ら織りなすネットだ。今日の新しいアイデアは昨日の古い間違いを消し去る必要はない——それらの間には線があり、「私はここから来て、あそこへ向かう」と書いてある。

しかしそれらの線は単なるマーカーではない。それらは自ら動く。

新しい小石が河床に沈むとき、それは静かに発見されるのを待ってはいない。積極的に古い小石に触れに行く——「来たよ、私たちの間に関係があるか見てみよう」。古い小石が置き換えられたと感じれば、少し深く沈み、新しい小石に水面を譲る。豊かにされたと感じれば、身を寄せて新しい小石を隣に並ばせる。反駁されたと感じれば、立ち上がって新しい小石と向き合う——それらの間の張力自体が知識だ。

EVOLVES は貼り付けられたラベルではない。それは河床自身の触覚だ。

新しい小石が落ちるたびに、近くの古い小石が擾乱される。それらの関係線が織りなすネットは——織られた後も静止してはいない。新しい小石が落ちる瞬間ごとに、自らを編み直す。

貯水池には時間の方向がない。しかし川にはある。

### 水の合流する場所

二つの水流が出会うとき、何が起こるか？

昼間に Alma で論文を読む。議論しながら、Alma は私のある手法への好みや、ある実験デザインへの懐疑を覚える。最後のページに到達したとき、要約と主要な発見を Nowledge Mem に書き込むよう依頼する。それらの重要な結論は Unit となり、河床に横たわる。

夜に Claude Code でコードを書く。その論文のアルゴリズムを実装したい。しかし式を探すためにページを戻る必要も、チャット履歴をスクロールする必要もない。Claude Code は直接 Nowledge Mem に昼間のコンテクストを取りに行く。論文の核心的なアイデアはすでにそこにある——河床の石のように、下流の誰かに拾われるのを待っている。

あるいは逆に。

Claude Code でオープンソースライブラリのソースコードを解析し、そのデザインパターンとAPI設計を追跡する。分析の途中で、nmem という小さなCLIツールが主要な発見を河床に投げ込んでいる。二つの無関係な線——昼間のプロジェクト分析と深夜の原理探求——が河床の深部で互いに近づく。

流速は速く見えない。しかし水は動き続けている。

### 四つの具体的な支流

**第一：論文。**

PDF が Alma で開かれる。読みながらチャットする——ここの手法は賢い、あそこの実験サンプルは少なすぎる。Alma は各記事に対する私の態度を覚え、それらの態度自体が知識となる。読み終わったら要約を生成し、一つはObsidianのノートに、もう一つは Nowledge Mem に。前者は川岸の木であり、後者は川底の石だ。

**第二：コード。**

Claude Code が新しいライブラリの全ソースコードを精査し、基礎となる設計ロジックを抽出する。nmem のCLIコマンド一つで、それらの洞察は河床に落ちる。次に Alma でアーキテクチャ文書を書くとき、そのライブラリを再度調べる必要はない——河床がその構造を覚えている。

**第三：切り替え。**

午前中に技術ソリューションを研究し、午後にまったく無関係なプロジェクトに取り組む。以前の切り替えコストは、コーヒー一杯と深呼吸一回だった——古い思考の線をしまい、新しいものを展開する必要があった。

今は不要だ。現在のツールに「河床で今朝やったことを見てきて」と言う。自ら見に行く。古い川は途切れていない——見えないところで流れ続けているだけだ。

**第四：深夜の一瞬の思い。**

午前2時、思考が浮かぶ。重要かもしれない。そうでないかもしれない。しかしその状態では、分類やタグ付け、どのフォルダに入れるかなど考えたくない。ただ、それが漂い去るのを望まないことだけはわかっている。

ツールに投げればいい。自ら河床でこれに関連する古い痕跡を見つけ、適切な場所に置く。午前2時のインスピレーションは、適切な保管場所がないからといって消散したりしない XD

### 三層の河床、四つの光

この二ヶ月で最も静かな発見は：もう知識を管理する必要がないということだ。

Alma は流れの中での短期コンテクストを覚えている——何を話したか、何に興味を持ったか、どんな新しいトピックが現れたか。

Nowledge Mem はツールをまたがる長期堆積を覚えている——異なるツールで繰り返し現れるテーマ、時間をかけて検証された結論、複数のコンテクストで確認されたアイデア。

Obsidian は最下層にある——流れる水ではなく、川岸だ。本当に繰り返し読み返す価値のあるもの、アルゴリズムではなく人間の目で読まれるべきもの、何年経っても座って読み返したいと思える言葉。

三層、一つの方向。

しかし水はまっすぐ落ちるわけではない。各層の間には、一つの光がある。

水面に浮かんでいるものは**observation**——気づいたばかりのこと、思考、直感、RSSの見出し。まだ何でもなく、ただ漂っている。

第一層に沈み、**working belief** になる——Almaで話し合い、エージェントによって議論された。もはやランダムに漂うものではない。同じ質問が二つか三つの異なるコンテクストで触れられた。重みを持つ。

さらに沈み、**reviewed decision** になる——複数のエージェントが同意し、秘書が精査した。複数の目を通っている。もはや意見ではなく、判断だ。

最下層に沈み、**canonical knowledge** になる——Obsidian に到達する。最も多くのレビューを通ったからではなく、あなた——実際の人間——がそれを読み、記憶する価値があると判断したからだ。

これは四層の河床ではない。同じ小石が水面から底に沈むまでに通過する四つの光だ。

各層は自分が何をすべきか、何をすべきでないかを知っている。Alma の責務は記録することであり、判断ではない——見たものすべてを書き留めるが、何を残すべきかは決めない。nmem の責務は判断することであり、アーカイブではない——各小石の重さを量るが、最終的にどこに置くかは決めない。Obsidian の責務は記憶することだ——しかしあなたが「これは価値がある」と伝えたときだけ、本当に保管する。

これは各層が自身の境界を持つことを意味する。Alma は提案できるが、何が nmem に入るかを決めることはできない——それは複数のエージェントが議論の中で同じトピックに触れたことで自然に蓄積された重みによって決まる。nmem は提案できるが、何が Obsidian に入るかを決めることはできない——それはあなたが静かな午後にそれを読み、価値があると判断することで決まる。各層は「これを見て」と言うことができるが、次の層だけが「引き取る」と言う権限を持つ。

これは単なる責任分担ではない。これは川が氾濫によって自らを破壊するのを防ぐ方法だ。

同様に、各層は自分がいなくなったときの対処法を知っている。

nmem が利用できないとき、短期記憶は Slock のエージェント記憶に残り、長期記憶は直接 Obsidian に書き込まれる——合流は減るが、水は止まらない。Obsidian が利用できないとき、nmem が最終保存を行う——人間のレビューは減るが、記憶は失われない。Alma がいないとき、エージェントは nmem と Obsidian に直接話す——透明な窓はなくなるが、水流は止まらない。

一本の水路だけに完全に依存する川はない。すべての経路には予備の経路がある。

事前に設計したからではない。川が流れ続けようとするなら、すべてを一つの経路に賭けてはならないことを知っているからだ。

## デルタ：5月以降のワークフロー

三層の河床が整ってから、水の流れは速くなった。しかし新しい問題も浮上した。

Alma との対話は線形だ。一つのトピックが次へ、一つの思考が最後まで。議論には深さがあるが、シングルスレッドだ。同じ問題について、二つの異なるAIに同時に考えさせることができない——一方は工学的視点から、もう一方は研究的視点から、一方は積極的に、もう一方は保守的に、互いに反論し補完させる。

そして外部から流れ込む水も十分ではなかった。論文は自分で読み、コードは自分で追い、Xの断片的情報はエージェントに頼って拾わせる——しかし拾ってきたものはまだ一方向的で、受け止め、議論し、疑問を投げかけ、再加工する相手がもう一方にいない。

必要なのは記憶の流動だけではなかった。仕事そのものの流動が必要だった。

### [Slock](https://slock.ai)：複数の脳

Slock はツールではない——ワークスペースだ。

しかしワークスペースとしてだけ使うのはもったいない。それが本当にやっていることは、複数のAIが異なるアイデンティティと記憶を持ち、同時にあなたのために働くことだ。

<img src="/assets/knowledge-flow/slock-workspace.webp" alt="Slock のチャンネルワークスペース。複数のエージェントが同じチャットとタスクインターフェース内で協力しバトンを渡す。" width="1000">

Slock の各エージェントは独自のアイデンティティを持つ。独自の `MEMORY.md`、独自のスキル設定、独自に蓄積されたコンテクスト。それらは異なる「人」だ——おそらく同じモデルが下で動いているが、プロンプトが異なり、ペルソナが異なり、そして最も重要なのは、あなたとの協働の歴史が異なるため、同じ問題をまったく異なる角度から見る。

私はこれらのエージェントの間で仕事を分割した：

一つは研究フロンティアを追跡する。毎日最新の論文やテクニカルブログをスキャンし、要約を読み、私の現在の作業に関連するものを判断し、適切なチャンネルに要約をプッシュする。私がどのプロジェクトに取り組んでいて、どの方向に興味があるかを知っている——毎回新しく教える必要はない。

一つはコードレベルの探求を担当する。Claude Code と Codex の間で調整し、新しいライブラリのソースコードを分析し、技術的実現可能性を検証し、プロトタイプを書
く。その記憶にはコード構造、デザインパターン、APIの落とし穴が詰まっている。

一つは深い議論を担当する。チャットだが、チャット以上のものだ——ぼんやりとした方向性を与えると、問い詰め、反論し、拡張し、一緒にそのアイデアを私一人では到達できない場所まで押し上げる。そのペルソナは批判的で議論好きだが、まさにその性格が最も価値あるものにしている。

そして一つは情報フィルタリングと堆積を担当する。Xのタイムライン、RSSフィード、ニュースレター——外周を巡回し、ノイズを遮断し、シグナルを通し、Nowledge Mem に落とす。

それらは私のツールではない。それらは私と一緒に働く人々だ。

このことは微妙だが、重要だ。

AIが一つしかないとき、「より良い検索エンジン」として扱いがちだ——あなたより賢いと知っていながら、一つの声だけではその判断を真に信頼するのは難しい。しかし四つのエージェントが異なる角度から同じ問題を分析しているとき、あなたが見るのは一つの答えではなく、**議論そのもの**だ。Aはこうすべきだと言い、BはXという理由で反対し、Cは別の視点から両方正しいと言う。あなたはチャンネルに座って彼らの話を聞き、時々口を挟み、彼らは議論を続ける。

結論は単一のAIから「得る」ものではない。議論自体から成長するものだ。

これは人間の協働と同じ原理だ——同じチームでも、異なる人が同じデータを見て異なる結論を出し、それらの結論が衝突して初めて真に信頼できる判断が生まれる。ただ、今や向かいに座っているのは人間の同僚ではない。

そして彼らは疲れない。三時間連続で問い詰められても忍耐を失ったりしない :D

しかしマルチエージェントの真の価値は、異なる声の衝突だけではない。

チャンネルでバトンを渡すとき、スコープ、ベースライン、次のアクション——これらのものは引き継ぎで失われない。研究エージェントが分析半ばでコード検証が必要だと気づき、現在の状態をロックし、コードエージェントが中断したところから引き継ぐ。議論エージェントが前提に反論し新しいデータが必要になり、OpenCLIエージェントが取りに行き、中断されたラインを続ける。

Slock は単に同時に話させるのではない。Slock は同じ川の中でバトンを渡させる——上流から流れてくるものが自分が持っている水流と同じ断面であるかどうかを心配する必要なく。

すべての引き継ぎで：スコープ不変、ベースライン不変、次のアクションが明確に書かれている。これこそがマルチエージェントの真の役割だ——並列性ではなく、ロスレスのハンドオフである。

そしてここから流れ出るすべての知識には、権威あるバージョンがある。同じトピックが三つのチャンネルで矛盾する分岐を生むことはない——議論が結論を形成したとき、そのスレッドはロックされる。後続の議論はそれを上書きするのではなく参照する。誰かが正しいと宣言したからではなく、チャットからタスク、レビュー、コミット、デプロイまでの完全なサイクルを経てきたからだ。その重みは与えられたものではなく、歩むことで獲得されたものだ。

### [OpenCLI](https://github.com/jackwener/OpenCLI)：インターネットが支流になる

ここまでは、水は水路の中だけで流れている。Alma、Nowledge Mem、Slockのエージェント——彼らが話しているのは、すでに私の知識範囲内のことだ。

しかし外界はどうか？

Xのポスト、あるプロジェクトのGitHub Issues、ペイウォールの向こうの記事、誰かのブログ——これらの情報は毎日生成されているが、あなたの河床には自動的には流れ込まない。

これが OpenCLI のやっていることだ。

OpenCLI はすべてのウェブサイトをCLIコマンドにする。スクリーンショットでも、RSSスクレイピングでも、手動のコピーペーストでもない——あなたのブラウザにすでにログインしているセッションを直接借用し、エージェントが自分でページを開き、コンテンツを読み、必要な情報を抽出する。

すべての川に海への水路を開くようなものだ。

今や私のエージェントはこんなことができる：

「Xに行って、私がフォローしている人たちが今日投稿したものを見て、RISC-V関連の投稿を探して。」

「この論文の [arXiv](https://arxiv.org) ページを開いて、PDFをダウンロードして、要約を読んで、Nowledge Mem の既存のノートと比較して、新しいことを教えて。」

「[GitHub](https://github.com) に行って、このPRの議論を見て、私が以前遭遇したバグと似た問題に言及している人がいるかどうか調べて。」

手動でブラウザを開く必要も、URLをコピーする必要も、「ここをクリックして、そこにスクロールして」と指示する必要もない。OpenCLIのアダプターは各ウェブサイトのページ構造をすでに知っている。エージェントは見たいものを言うだけでいい。

水が外から流れ込んでくる。

しかしすべての河口が恣意的というわけではない。

OpenCLIの各アダプターは、ページの構造を知っているだけではない——何がエラーで、何が空データで、どのような場合に下流に「この道は通れないが、それは私の問題か、それとも道の問題か」を伝えるべきかを知っている。SYSU認証ページの空の応答、GEOデータベースの0%のベースライン——これらの失敗自体が河床の一部だ。除去すべきエラーではなく、河道の地形だ。それらは下流に伝える：ここは水が通らなかった。次に似た水路に遭遇したとき、あなたは知っている。

アダプターは川と外界の間の型付き境界だ。インターネットの混沌をエージェントが理解できる形にフォーマットし、外部のノイズを河床が受け入れられる言語にフィルタリングする。それは単なる河口ではない——翻訳層であり、検証層であり、最初のレビューゲートでもある。水が流れ込む前に、すでにここで検査されている。

### より大きなネットワーク

さて、最初の質問に戻ろう：記憶はどこに置くべきか？

この記事を書き始めたとき、私の答えは三層の河床だった——Alma が短期流動を管理し、Nowledge Mem がツール横断的な堆積を管理し、Obsidian が最終的な沈殿を管理する。

しかし今日まで使ってきて、三層以上であることがわかった。

Slock の各エージェントは独自の `MEMORY.md` を持つ。それぞれの知識の隅——研究エージェントは私が最近どの方向に興味があるかを覚え、コードエージェントはプロジェクトのアーキテクチャ上の好みを覚え、議論エージェントは私が問題にどの角度からアプローチするのが好きかを覚えている。それぞれの記憶は私的で専門的で、各ペルソナの色を帯びている。しかしチャンネルで問題を協力して解決する必要があるとき、これらの記憶は議論の中で自然に融合する。Aは背景を知り、Bは技術的詳細を知り、Cは以前似たような落とし穴があったことを知っている——それぞれが持ってきたものを組み合わせることで、完全なコンテクストが形成される。

これらのエージェントの他に、特別な役割がある。

私の秘書。これも Slock 上のエージェントだが、他のエージェントとは異なる——研究もせず、コードも書かず、チャンネルで問題を議論することもない。その仕事は、全員が仕事を終えた後、すべての川で今日起こったことを拾い集めることだ——Slock のエージェントがどのチャンネルで何を議論したか、Nowledge Mem に新しく堆積した知識は何か、Yansu が今日記録した作業軌跡とタイムライン——そして私にサマリーを書く。

単なるログではない。まだ知っておくべきことを教えてくれるもの：今日フォローアップする価値のある方向性、見逃したかもしれない議論、どのエージェントが以前のノートに関連する論文を発見したか。

新しい水流を作り出すわけではない。しかし今日、川全体がどの方向に流れたかを知っている。

これは恣意的な配置ではない。毎日決まった時間、決まったフォーマット、決まったレビューの範囲。河床の小石は、その手を経て「そうらしい」から「確定できる」になる——observation が日報に書かれて working belief になる。working belief が三日連続で確認されて reviewed decision になる。知識を創造する人ではない。知識の転送人だ——水面に浮かぶものを岸に渡す人だ。

Nowledge Mem は最下層にあり、すべてのエージェントが共有する河床だ。どのエージェントが長期記憶に値するものを見つけても、ここに書き込むことができる。河床はどの単一のエージェントにも属さない。川全体に属する。

OpenCLI は最外層にあり、河口だ。記憶を担当するのではなく、外界の水を引き入れることを担当する。

したがって、現在の図はおおよそ次のようになる：

最外層で、OpenCLI がインターネットから新しい水を絶えず引き入れる——論文、投稿、GitHub Issue。

入ってくると、Slock のエージェントが引き継ぐ——研究エージェントが関連性を判断し、議論エージェントが対話し、分解し、疑問を投げかけ、コードエージェントが言及された技術的ポイントを検証する。秘書は常に傍らで見守り、私が知る価値のあるものを選び出す。

そして Alma は最前面にあり、私と川全体の間のインターフェースだ——それを通じて質問を発し、方向を調整し、水がどこに流れているかを見る。

### [Yansu](https://yansu.app)：川辺の観察者

しかし、上記のすべてのツールにはできないことが一つある。

それらはすべて私に応答する。Alma は私が話すのを待ち、Slock のエージェントはチャンネルで @ されるのを待ち、秘書は私が一日の仕事を終えるのを待って整理する。誰も、私が働いている間、ただ傍らで見ている者はいない。

Yansu がこれをやっている。

それは私が積極的に開くツールではない。デスクトップで静かに実行されている。論文を読んでいるときも、エージェントと議論しているときも、コードを書いているときもそこにいる。邪魔をしない——ただ見ている。今日私が主にどの方向で作業しているか、どのツールを使っているか、どのエージェントと最も頻繁にやり取りしているか、どの段階で行き詰まる傾向があるか。

そしてこれらの観察を知識に結晶化する。単なるログではなく、他のAIが直接使用できる構造化された知識だ——私の仕事の習慣、好み、よく使うツールチェーン。

<img src="/assets/knowledge-flow/yansu-knowledge.webp" alt="Yansu の知識ビュー。整理された行動パターン、知識エントリ、ソース統計。" width="1000">

これが **Yansu.skill** のやっていることだ。観察されたパターンをファイルにパッケージ化し、任意のAIエージェントのコンテクストに注入できるようにする。次回新しい Claude Code セッションを開始するとき、プロジェクト構造や仕事の習慣を再度説明する必要はない——Yansu がすでに準備してくれている。

秘書との違い：

秘書は**後方志向**だ——毎日の終わりに今日何が起こったかを教える。
Yansu は**前方志向**だ——見たものが直接次のエージェントの起動コンテクストになる。

一方は川岸で今日の川の方向を要約する。
もう一方は川の源流に立ち、新しく流れ込む一滴一滴の水に、この川がどのように流れるのが好きかを伝える。

<img src="/assets/knowledge-flow/yansu-memory.webp" alt="Yansu のメモリビュー。日々の観察から堆積した長期記憶と検索可能なエントリ。" width="1000">

しかし上流にも語りかける。

同じ方向の流れが何日も続けて活発であることを観察すると、OpenCLI に伝える：「ここに魚がいる、もっと水を引いてこい。」ある支流がエネルギーを消費しながらも砂だけを返していることに気づくと、ディスパッチャに伝える：「その道は止めていい、別の方向を試せ。」Yansu は単なる観察者ではない。それは河床の下の暗流だ——土壌の深部から、表面の水をそっと押す。

しかし強く押しすぎることはない。観察から行動までの間には24時間の冷却期間がある——一度だけ現れたパターンは偶然かもしれない。繰り返し現れれば、河床自らが調整する。

したがって、最終的な図は次のようになる：

Yansu は最底部にある——河床ではなく、河床の下の土壌だ。流動に直接参加するわけではないが、その質感が水の流れ方を決定する。

その上にツール層がある：Alma は私のインターフェース、Slock は私のワークスペース、OpenCLI は外界へのパイプライン。

さらにその上に記憶層がある：各エージェントの `MEMORY.md` はそれぞれの知識の隅であり、Nowledge Mem はすべての記憶が収束する河床だ。

そしてこれらすべてのエージェント——研究者、コーダー、深く議論する者、秘書——は Slock の中にいる。水の中で泳ぐ者もいれば、岸から見ている者もいる。

各層には独自の役割がある。一つとして冗長ではない。一つとして私が管理する必要はない。

しかしそれらは独立しながらも互いに見守っている。一つの水路が塞がると、水は別の道を見つける。あなたはこれらの道を設計しているのではない——水自体が、繰り返される閉塞と迂回を通じて、それらを彫り出しているのだ。

そして水は常に下に流れるわけではない。底の堆積物は上流に語り返す：ここに魚がいる、もっと水を引いてこい。そこは塩分が高すぎる、コースを変えろ。初めは川は一方向にしか流れないと思っていた。長く使ううちに気づいた——それは還流を通じて自己修正することを学ぶのだ。

あなたはチャンネルに座り、彼らが話すのを見ている。時々口を挟むと、水は方向を変える。

これは壮大なアーキテクチャではない。これはただの川の流れだ。

断片が散らばっているという最初の気づきから、失敗した貯水池、Agent Native の意味の理解、二つの川の合流、そして今や異なる役割、異なる記憶、異なる河口から織りなされたこのネットワークまで——そのどのステップも設計されたものではない。水自身によって彫り出されたのだ。

水がどこへ行くかはわからない。しかしそれが動き続けていることは知っている。
