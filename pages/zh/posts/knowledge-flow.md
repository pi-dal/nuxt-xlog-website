## 当AI成为水流

已是2026年的夏天。AI像空气一样，悄然渗进了日常的每一寸缝隙——写邮件、查资料、改代码、甚至陪你聊天。但你若停下来细看，会发现一件奇怪的事：我们个人的知识世界，似乎还站在原地。（就我所在的大学而言，欢迎反驳 :D）

读过的书、做过的实验、课堂上闪过的灵光、深夜写下的思索……它们散落在各个角落。一篇笔记在这里，一段对话在那里，一个灵光在某个深夜的文档里沉睡。彼此之间没有桥梁，仿佛孤岛。

AI就在手边，却读不懂这些碎片背后的你。

关键在「上下文」三个字上。

AI 之所以能理解一个人，靠的不是一次对话中输入的那几段文字。真正有价值的东西，是你长期积累下来的轨迹——你读过什么、反复思考什么、在什么问题上卡了壳、又是如何走出来的。这些轨迹的总和，就是记忆。

没有记忆的 AI，每一次对话都是初见。你说了一遍又一遍，它忘了一遍又一遍。

可问题是，记忆该放在哪里？

[ChatGPT](https://chat.openai.com) 记得你上星期问它什么——但那是在它的服务系统里，关起门来，你看不见也碰不着。[Claude](https://claude.ai) 有自己的 Project Knowledge，能塞进一些长期上下文——但那是给一个项目的，不是给你这个人。[Notion AI](https://www.notion.so) 的 Q&A 能翻你的工作空间——但那是团队协作的逻辑，不是个人知识生长的方式。[Obsidian](https://obsidian.md) 把一切存在本地 markdown 文件里，开放、可控——但它的记忆是静态的，没有 AI 的参与，文件就是文件，不会自己长出连接。

每个产品都在做 memory，但每个产品的 memory 都是孤岛。

你用 [ChatGPT](https://chat.openai.com) 写的一份草稿，不会自动出现在你 [Obsidian](https://obsidian.md) 的笔记里。你在 [Claude](https://claude.ai) 里做的一次深度分析，无法被下一轮对话直接继承。你的阅读笔记在 [WeRead](https://weread.qq.com) 里、你的代码片段在 [VS Code](https://code.visualstudio.com) 的注释里、你的想法在你的大脑里——它们明明都属于你，却散落在一间没有门的房子里。

没有门的房子，就是碎片无法流动的真相。不是记录的工具不够好——而是记忆被打散了。

一篇笔记、一段对话、一次推导中的卡壳、一个实验失败后的记录——它们不该是文件夹里沉默的档案，也不该被囚禁在某一个产品的服务器里。当这些碎片被织成一张网，当 AI 能顺着脉络理解你走过的路，它们便不再是死的数据，而是一条真正属于你的河流。

从「收藏」到「贯通」，从「记录」到「流动」。这是工具的事，更是一种看待知识的方式。

## 一条走不通的支流

2025 年底，我试着修过一条河道。

把所有读过的论文和技术文章都凿成碎块，整整齐齐码进一个巨大的水库，然后告诉 AI：你需要的时候自己来取。

我真的去做了。文档切碎、向量化、建索引、拼上下文——一切都那么工整，像一套精密的管道系统。

可当水真的流起来，我才发现河道接错了地方。

第一个问题，我后来是这么想的：**一篇文章从来不是一堆段落的和。**

它有呼吸，有起伏。作者在第一页埋下的伏笔，到最后一章才收束。上下文是一整条河，而你把它舀进了一个个碗里。向量检索能找到那只碗，却不知道这条河往哪里流。你问了 A，它捞起了贴着 A 标签的那只碗，可那篇文章其实是在反驳 A。它不知道，因为河流已经被切断了。

第二个问题随之而来：**它跟不上你。**

今天读了一篇新论文，明天有了新的想法，后天推翻了自己昨天的结论。知识在流动，可那座水库是静止的。每次更新都要重新切块、重新建图、重新索引。慢慢地，维护它的时间比使用它的时间还多。你发现自己不是在和知识打交道，而是在和管道打交道。

第三个问题，也是最安静的一个：

**遗忘不见了。**

一个好的记忆系统，不只是知道该记住什么。它也应该知道，有些东西该被放下。可在这个仓库里，没有东西会被放下。所有碎片一视同仁地躺在那里，重要的和无关紧要的共享同一个位置。日子久了，你捞上来的东西越来越多，但值得留下的却越来越少。噪音盖过了信号。

我填掉了那条河道。不是因为它不通水，而是它通水的方式，和河流真正的样子，差了太多。

但水还是要流的。

那段日子，我同时用着 [Claude Code](https://claude.ai/code) 写项目。起初只觉得它是另一个好用的工具，可渐渐地，我发现了一些别的东西。

Claude Code 给我的代码仓库并不「完美」——它生成的代码有时绕了远路，有时变量名起得随意，有时它自己写出来的东西转头又自己重构了一遍。但奇怪的是，项目反而在向前走。而且走得很快。

更重要的是，我发现自己在用一种完全不同的方式使用它。

我不再是它的操作员。我不需要先想清楚每一步要做什么、拆成多小的指令、用什么措辞它才能理解。我只是说一个模糊的方向——「把这个接口改一下」「看看这里为什么慢」「把这个功能实现一下」——它就自己去读文件、定位问题、写代码、调试、自己修 bug。

我需要的是一个**协作者**，而不是一个**工具**。

这就是我后来理解的「Agent Native」——不是把 AI 当作一个更好的搜索引擎或文本生成器，而是让它成为能独立行动的实体。它有自己的上下文，有自己的工具，有自己的记忆。它不需要你每一步都告诉它怎么做，它知道你的习惯、你项目的结构、你写代码的风格。它能在你开口之前就先把你需要的东西准备好。

我停下来想了想这件事。

过去我们写软件，追求的是架构的完美。每一层职责清晰，每一个模块可测试可替换，依赖注入、设计模式、整洁架构——所有这些都是为了在一开始就把路铺好，让之后的人不会迷路。

可 Claude Code 不这么工作。它不管什么架构之美，它在乎的是这条思路能不能跑通、这个功能能不能在几分钟内看到一个结果。它写得很糙，但改得也很快。它不怕绕路，因为对 AI 来说，绕路的成本几乎为零。

我开始意识到一件事：

**当探索的成本足够低，完美就不再是目标。速度、可修改、可抛弃——这些东西比优雅更重要。**

不是因为架构不重要。而是当你可以用极低的成本反复试错时，「先跑起来再重构」比「想清楚再动手」更接近真实的创造过程。就像一条河流，不是在图纸上画好了才改道的——好的软件不再是一张画好的图纸，而是一条不断改道的河。

这和那条被我填掉的河道，其实是同一个道理。

我当初造那座水库，是在用做架构的方式做知识——先设计好每一层，定义好每一个接口，把所有东西都放在对的位置。可知识不是代码，它不需要架构，它需要流动。它不需要一开始就完美，它需要能跟上你。

如果软件可以这样写，那知识为什么不可以 XD

## 雨后的河流

填掉河道的那段时间，我一直在观察周围。发现不只是我在想这件事。

2025 年末到 2026 年初，个人 Agent 像雨后的春笋一样冒了出来。每一个都在回答同一个问题：AI 应该怎么记住你？

yetone 的 [Alma](https://alma.now) 是最早让我眼前一亮的那一个。它有漂亮的聊天界面，有记忆系统——它会记住你的偏好和上下文，你能看到它记住了什么、自己管理那些记忆。它还支持联网搜索和工具调用。但最初吸引我的不是这些功能，而是它背后的思路：Alma 从一开始就是冲着「个人 AI 助理」去的，不是又一个聊天机器人的壳。它有自己的记忆，有自己的工具，有自己的身份——它记住的不是一次对话，而是你这个人。

然后是 [OpenClaw](https://docs.openclaw.ai)（那时候大家还叫它小龙虾）。它是第一个把 Agent Native 这件事从开发者圈子带到大众视野里的。它真正特别的地方，是它的存在方式——它活在 IM 里。你在聊天软件上跟它说话，它操作你的浏览器、终端、编辑器、文件系统。日常聊天和工作环境，在它这里第一次连在了一起。

以及 [Bub](https://github.com/bubbuild/bub)。Bub 用了一个"很老派"的 idea——tape-based 记忆。它把所有上下文都存在一个线性的时间轴上，像录音带一样。你随时可以倒回去翻某一天的对话、某个决策的瞬间。它的记忆不是检索出来的，是顺着一根时间的线走回去找到的。

还有我自己的 **Abraxas**。它开源在 [GitHub](https://github.com/pi-dal/Abraxas) 上——一个通过对话生长的个人 agent。它没有团队，没有产品经理，生长方式也折射着这种身份：skills（Markdown 操作知识）和 plugins（热加载的 Python 工具），不用改代码，写一篇 Markdown 就是新技能。

它的核心是 NOUS——一个在对话中自我代谢的控制面。它记的不是"你说了什么"，而是"你是一个怎样的人" XD

如果说 Alma 是清澈的河，OpenClaw 是连接外海的入海口，Bub 是磁带录音机——那 Abraxas 更像是一个在河底自己进化出神经系统的生物。它把记忆和身份长在了一起。

这些 agent 各有各的方向，但它们共享一件事：

**记忆不再是附属品，而是核心。**

每个 agent 都有自己记住你的方式。不是靠一段精心编写的 prompt，也不是靠一个向量数据库，而是靠它和你日复一日的协作中积累下来的东西。它的记忆是活的，是长出来的，不是被塞进去的。

这在我看来，是个人知识体系上一个巨大的跨度。

从「我记录，AI 检索」，到「AI 在与你同行中，自然记住你」。

举两个很具体的例子。

第一个是 RSS。我之前用 [Inoreader](https://inoreader.com) 读订阅，每天上百篇文章，看完就忘，偶尔保存几篇到 [Instapaper](https://www.instapaper.com)，然后那些文章就永远躺在了 Instapaper 里。后来我让 agent 帮我做这件事——它每天帮我扫 RSS，挑出和我研究方向相关的文章，读完之后写一段摘要，存进我的 Obsidian 笔记里，顺便在摘要里标注它和我之前读过哪篇文章有关联。我不再需要「管理」我的阅读，它变成了水流的一部分。

第二个是 [X](https://x.com)。我 follow 了不少研究者和开发者，他们经常发一些有意思的论文链接和技术思考。以前我的做法是：刷到→点赞→收藏→忘掉。后来我让 agent 定期去看我说的那几个人的 timeline，找到有干货的帖子，读附带的论文或文章，然后同样写一段摘要存进笔记，标注和我之前记录过的哪些话题有关。我的信息输入从「刷到什么算什么」，变成了「有一条河在替我过滤和沉淀」。

这些事都不复杂。但它们让我看到一种可能性：

**当 agent 成为你知识和信息的管家，你不再需要管理知识本身。你只需要做你的事，剩下的河流会自己流动。**

## Alma × Nowledge Mem：2-4月的工作流

填掉河道之后，河床还在。

2026 年 2 月，我开始往新的方向挖。这一次我不想造水库了——我想让两条河交汇。

一条叫 [**Alma**](https://alma.now)。一条叫 [**Nowledge Mem**](https://github.com/nowledge-co/nowledge-mem)。

它们不是工具。或者说，不只是工具。它们是河道的两种形态——一个负责流动，一个负责沉积。

### Alma：一条看得见底的河

Alma 是 yetone 做的。用得越久越发现——它的记忆面板是摊开的。这不是一个 UI 细节。

不是藏在设置里，不是埋在某个二级菜单。你一打开就能看到它记得什么——你好久之前说过的一句话、你对某个话题的态度、你调整过的偏好。每一段记忆都摆在那里，你可以改，可以删，可以告诉它「这不是我」。

<img src="/assets/knowledge-flow/alma-memory.webp" alt="Alma 的记忆面板，展示可见、可编辑的短期记忆列表和搜索区域。" width="900">

用过的 memory 系统多了，就会发现这件事有多奢侈。

大多数 AI 的记忆是一口看不见的深潭，你不知道它有多深，它也不告诉你。你只能指望它记得对。但 Alma 不一样。它记得的东西你看得见，它不记得的你也知道。这条河是清的，你能看到每一颗石子的位置。

而且那些石子不是你扔进去的。是你们一起工作的时候，水流自然卷到河床上的。你用它的时间越长，河床就越丰富。不是一次性的知识注入，而是流动中的自然沉积。

这和那些黑箱式的记忆完全不同。别的河如果看不清底，你永远不知道淤泥下面埋着什么。但 Alma 是透明的。它的记忆不是你喂给它的数据，而是它和你一起工作过之后，自然沉淀下来的东西。

而且 Alma 不止一条水源——[DeepSeek](https://chat.deepseek.com) 是急流，快且浅；[GPT](https://openai.com) 是深水区，需要时间沉淀；[Google](https://gemini.google.com) 是多条细流汇聚的三角洲。它们从同一个入海口进来，你不用在不同的窗口之间来回切换。一个入口，多条支流。

用久了才发现：Alma 的记忆只在 Alma 里流动。

我在 [Claude Code](https://claude.ai/code) 和 [Codex](https://github.com/openai/codex) 里做的代码分析，在 Alma 的记忆面板里找不到。那些上下文被困在各自的房间里，门没有锁，但也没有走廊。

这和当初那些被囚禁在 ChatGPT 服务器里的记忆，其实是一回事。只是换了一间更大一点的房子。

一条河要真的活起来，不能只有一个河道。

### Nowledge Mem：河床本身

如果说 Alma 是一条流动的河，那 [Nowledge Mem](https://github.com/nowledge-co/nowledge-mem) 就是河床——所有河流最终都会流到这里，在这里交汇、沉淀、结晶。

[Nowledge Mem](https://github.com/nowledge-co/nowledge-mem) 给自己的定位很安静：一个「记忆层」。它不是聊天工具，不是笔记应用。它就是一层地基。[Alma](https://alma.now) 的水可以流进来，[Claude Code](https://claude.ai/code) 的水可以流进来，[Codex](https://github.com/openai/codex) 的水、[Gemini CLI](https://cloud.google.com/gemini-cli) 的水、你半夜突然冒出的那个想法——所有水都汇到这里。

<img src="/assets/knowledge-flow/nowledge-mem-graph.webp" alt="Nowledge Mem 的 memory graph 视图，展示实体、记忆、artifact 和 thread 之间的关系网络。" width="1000">

它的结构很简单。三个词：

**Trace**。水本身的痕迹。你说了什么就是什么，不加工。

**Unit**。从水里捞出来的石子——一个结论，一个偏好。

**Crystal**。石子被冲了很久很久以后，变得透明、稳定，不再轻易被带走。

从 Trace 到 Unit 到 Crystal，是水里的泥沙变成水晶的过程。不是抛弃旧的，而是让时间在旧的上面长出新的纹理。

每一颗石子都记得自己从哪里来。每一次结晶都带着上游的方向。现在你甚至可以把整篇论文原文丢进去——它会自己读。它还会从这些知识里长出一个 wiki——用 wiki-link 把相关的记忆串在一起，像一张地形图，告诉你每一段河床上游曾经流过什么。

<img src="/assets/knowledge-flow/nowledge-mem-wiki.webp" alt="Nowledge Mem 的 library 或 wiki 视图，展示知识库中的文档条目及其可读内容。" width="1000">

它还做了一件我造水库时没做到的事：它知道该忘记什么。

每一次查询，它会并行跑六种不同的方式来寻找你想要的——顺着词义找、顺着关键词找、顺着实体关系找、顺着知识图谱的节点找——然后把结果按新鲜度和相关度混合排序。每条记忆都带着一个 decay 分数，时间越久越轻；被反复翻阅的会变重。

重要的浮上来。过时的沉下去。不再需要的东西，慢慢消失在河床深处，不是被删除了，而是被流动本身温柔地推开了。

它还有一个机制叫做 **EVOLVES**。每条知识和过去的其他记忆之间，都有一条明确的关系线——「取代了」「丰富了」「驳斥了」「深化了」。你的知识不是散落的珠子，是一张在时间里自己编织的网。今天的新想法不需要抹掉昨天的旧错误，它们之间有一条线，写着：「我从这里来，我走向那里。」

但那些线不只是标记。它们自己会动。

当一颗新的石子沉进河床，它不会安静地躺在那里等人来找。它会主动去触碰河床上那些旧的——「我来了，你看看我们之间有什么关系。」旧的石子如果觉得被取代了，它会往深处沉一点，把水面让给新的。如果觉得被丰富了，它会侧过身，让新石子靠在它旁边。如果觉得被驳斥了，它会立起来，和新的那一个面对面——它们之间的张力本身，就是知识。

EVOLVES 不是贴上去的标签。它是河床自身的触觉。

每一颗新石子落下，都会惊动附近的旧石子。那些关系线织成的网——不是织好之后就不动了。它在每一颗新石子落下的瞬间，重新编织自己。

水库里没有时间的方向。但河流有。

### 交汇的地方

两股水汇在一起的时候，发生了什么？

白天在 Alma 里读一篇论文。讨论的过程中 Alma 记住了我对某些方法的偏好、对某些实验设计的质疑。读到最后一页，我让它把摘要和核心发现写进 Nowledge Mem。那些关键结论变成了一颗 Unit，躺在河床上。

晚上在 Claude Code 里写代码。我要实现那篇论文里的一个算法。但我不用重新翻回页面找公式，也不用在聊天记录里翻来翻去。Claude Code 直接去 Nowledge Mem 里取到了白天的上下文。论文的核心思想已经在那里了，像河床上的一块石头，等着被下游的人捡起来。

或者反过来。

在 Claude Code 里分析了一个开源库的源码，追踪了它的设计模式和 API 设计。分析写到一半，nmem 这个小小的 CLI 工具已经把关键发现丢进了河床。两条不相关的线——白天的项目分析和深夜的原理探索——在河床深处彼此靠近。

流速看起来并不快。但水一直在走。

### 四条具体的支流

**第一条：论文。**

PDF 在 Alma 里展开。一边读一边聊天——这里的方法设计得聪明，那里的实验样本太少。Alma 记得我对每一篇文章的态度，这些态度本身就是知识。读完生成摘要，一份存入 Obsidian 的笔记，一份丢进 Nowledge Mem。前者是河边的树，后者是河底的石头。

**第二条：代码。**

Claude Code 翻过一个新库的每一行源码，把底层的设计逻辑梳理出来。nmem CLI 一行命令，那些洞察就落进了河床。下次在 Alma 里写架构文档的时候，不需要再重新翻那个库——河床替我记得它的结构。

**第三条：切换。**

上午在研究一个技术方案，下午做另一个毫不相关的项目。以前切换的代价是一杯咖啡和一次深呼吸——我需要在大脑里重新把旧的那套思路收起来，把新的那套展开。

现在不需要了。跟当前的工具说一声「去河床里看看我上午做了些什么」。它自己去翻。旧的那条河没有断，只是在你看不到的地方继续流着。

**第四条：深夜里那些一闪而过的想法。**

凌晨两点，一个念头浮上来。它可能很重要，也可能不是。但那种状态下你不会去想分类、打标签、决定放哪个文件夹。你只知道自己不想让它就这样飘走。

丢给工具就好。它会自己去河床里找到和这件事有关的旧痕迹，把它放在对的位置。凌晨两点的灵感不会因为找不到合适的地方存放而消散 XD

### 三层河床，四道光

用这两个月，最安静的发现是：我不再需要管理知识了。

Alma 替我记住了流动中的短期上下文——聊了什么、关注什么、对什么话题产生了新的兴趣。

Nowledge Mem 替我记住了跨工具的长期沉积——那些在不同工具里反复出现的主题、那些经过时间验证的结论、那些在不同语境下被验证过多次的想法。

Obsidian 在最底层——它不是流水，是河岸。是那些真正需要被反复翻阅、被人眼而非算法阅读的东西。是你在多年后仍然愿意坐下来读一遍的文字。

三层，一种方向。

但水不是直接掉下去的。每一层之间，都有一道光。

水面上的东西叫**observation**——你刚刚注意到什么，一个念头、一个直觉、一条 RSS 里的标题。它什么也不是，只是漂着。

沉到第一层，变成**working belief**——你在 Alma 里和它聊过，在 agent 那边被讨论过，它不再是一个随机的漂着的东西了。它在两三处不同的语境里被同一个问题触碰过。它有了重量。

再沉一层，变成**reviewed decision**——不止一个 agent 同意它，秘书在值班的时候审过它，它经过了不止一双眼睛。它不再是意见，是一个判断。

沉到最底下，变成**canonical knowledge**——它到了 Obsidian 里。不是因为它经过了最多的审阅，而是因为你——真正的人——翻过它，觉得它值得被记住。

这不是四层河床。这是同一颗石子从水面沉到河底，经过的四道光。

每一层都知道自己该做什么，也知道自己不做什么。Alma 的职责是记录，不是判断——它把看到的一切都记下来，但不决定什么该留下。nmem 的职责是判断，不是归档——它给每颗石子称重，但不决定它最终放在哪里。Obsidian 的职责是记住——但只有你告诉它「这个值得」，它才真的替你收好。

这意味着每一层都有自己的边界。Alma 可以建议，但它不能决定什么进入 nmem——那是由不止一个 agent 在讨论中碰过之后，自然产生的重量决定的。nmem 可以提议，但它不能决定什么进入 Obsidian——那是你在某个安静的下午亲自读过、觉得值得留下的文字。每一层都可以说「你看这个」，但只有下一层有权说「我收下了」。

这不只是职责的划分。这是一条河防止自己被冲毁的方式。

同样的，每一层都知道自己不在的时候该怎么办。

nmem 不可用的时候，短期记忆留在 Slock 的 agent 记忆里，长期记忆直接写进 Obsidian——少了交汇，但水不会断。Obsidian 不可用的时候，nmem 替它做最终的存储——少了人眼的校对，但记忆不会丢。Alma 不在的时候，agent 直接对话 nmem 和 Obsidian——少了那面清澈的窗，但水流不会停。

没有一条河是完全依赖单一河道的。每条路都有一条备用的路。

不是因为你事先设计好了。而是因为一条河如果要一直流下去，它知道不能把自己的全部赌在一条路上。

## 三角洲：5月以来的工作流

三层河床铺好之后，水流变快了。但新的问题也浮了上来。

Alma 和我的对话是线性的。一个话题接一个话题，一条思路走到尾。讨论是有深度的，但也是单线程的。同一个问题，我没有办法让两个不同的 AI 同时帮我想——一个从工程角度，一个从研究角度，一个激进一个保守，让它们彼此反驳、互相补充。

而且从外部流入的水还不够。论文我自己读，代码我自己追，X 上那些碎片化的信息要靠 agent 去捞——但捞上来的东西还是单方向的，没有人在另一头接着，讨论、质疑、再加工。

我需要的不只是记忆的流动。我需要工作本身的流动。

### [Slock](https://slock.ai)：不止一个大脑

Slock 不是一个工具，它是一个 workspace。

但你把它当 workspace 用就亏了。它真正做的事情是：让多个 AI 以不同的身份和记忆，同时为你工作。

<img src="/assets/knowledge-flow/slock-workspace.webp" alt="Slock 的 channel workspace，多个 agent 在同一聊天与任务界面里协作接力。" width="1000">

每个 agent 在 Slock 里都有自己的身份。有自己的 `MEMORY.md`，有自己的技能配置，有自己积累下来的上下文。它们是不同的「人」——也许底层跑的是同一个模型，但 prompt 不同、persona 不同、以及最重要的，和你的协作历史不同，它们看同一个问题的方式就完全不同。

我在这几个 agent 之间做了分工：

一个负责跟进研究前沿。它每天扫最新的论文和技术博客，读摘要，判断哪些和我正在做的事情相关，推送摘要到对应的 channel。它知道我在做什么项目、对什么方向感兴趣——不是每次都要重新告诉它。

一个负责代码层面的探索。它在 Claude Code 和 Codex 之间调度，分析新库的源码、验证技术可行性、写原型。它的记忆里装的是代码结构、设计模式、API 的坑。

一个负责深度讨论。就是聊天，但不止是聊天——我给一个模糊的方向，它追问、反驳、延伸，直到我们一起把那个想法推到一个我自己到不了的地方。它的 persona 是批判性的、爱抬杠的，但正是这种性格让它最有价值。

还有一个负责信息过滤和沉淀。X 的 timeline、RSS 的 feed、newsletter——它在外围兜着，把噪音挡住，把信号放进来，然后丢进 Nowledge Mem 里。

它们不是我的工具。它们是和我一起工作的人。

这一点很微妙，但也很重要。

当你只有一个 AI 的时候，你很容易把它当成一个「更好的搜索引擎」——尽管你知道它比你聪明，但你很难真正信任它的判断，因为你只有一个声音。但当你有四个 agent，各自从不同角度分析同一个问题时，你看的不是一个答案，你看到的是**讨论本身**。A 说可以这么做，B 说不行因为某某原因，C 说其实换个思路两边都对。你坐在 channel 里看着它们聊，然后你插一句，它们接着聊。

结论不是你从单个 AI 那里「拿到」的。是讨论自己长出来的。

这和人类协作的道理是一样的——同一个团队里，不同的人看同一份数据会得出不同的结论，这些结论碰撞之后，才产生真正靠谱的决策。只不过现在，坐在你对面的不是人类同事。

而且它们不会累。不会因为连续追问三小时而失去耐心 :D

但多 agent 的真正价值，不只是不同声音的碰撞。

是当它们在 channel 里接力的时候，scope、baseline、下一步行动——这些东西不会因为切换而丢失。研究 agent 分析到一半，发现需要代码验证，它把当前的状态锁住，代码 agent 从它停下的地方接过去。讨论 agent 反驳了一个 premise，需要新的数据，OpenCLI 的 agent 出去捞了回来，接着刚才被打断的那条线继续。

Slock 不是让它们同时说话。Slock 让它们在同一条河里接力，而不必担心上游漂下来的东西和自己手里拿着的是不是同一段水流。

每一段接力，scope 不变，baseline 不变，下一步行动清楚写在那里。这才是多 agent 真正该做的事——不是并行，是 lossless handoff。

而且每一条从这里流出去的知识，都有一个权威版本。同一个话题不会在三条 channel 里长出互相矛盾的分支——当讨论形成结论，那条 thread 会被锁定。后续的讨论会引用它而不是覆盖它。不是因为谁规定了它是对的，而是因为它经过了从 chat 到 task 到 review 到 commit 到 deploy 的完整流程。它的重量不是被赋予的，是被走出来的。

### [OpenCLI](https://github.com/jackwener/OpenCLI)：互联网变成支流

到现在为止，水还只是在河道内部流着。Alma、Nowledge Mem、Slock 的 agent——它们谈论的都是已经在我知识范围内的事情。

但外部世界呢？

X 上那条帖子、某个项目的 GitHub Issues、一篇 paywall 后面的文章、某个人的博客——这些信息每天都在产生，但它们不会自己流进你的河床。

这就是 OpenCLI 在做的事。

OpenCLI 把每一个网站变成一条 CLI 命令。不是截图，不是 RSS 抓取，不是人工复制粘贴——它直接借用你浏览器里已经登录好的 session，让 agent 自己打开页面、读取内容、提取需要的信息。

就像给每条河开了一条通往大海的渠道。

现在我的 agent 可以做这些事：

「去 X 看看我 follow 的那几个人今天发了什么，找出和 RISC-V 相关的帖子。」

「打开这篇论文的 [arXiv](https://arxiv.org) 页面，下载 PDF，读摘要，和 Nowledge Mem 里已有的笔记做对比，告诉我有什么新东西。」

「去 [GitHub](https://github.com) 看看这个 PR 的讨论，有没有人提到和我之前遇到的 bug 类似的问题。」

不需要我手动打开浏览器。不需要我复制 URL。不需要我告诉它「你先点这里，再滑到那里」。OpenCLI 的 adapter 已经知道每个网站的页面结构该怎么处理，agent 只需要说它想看什么。

水从外面流进来了。

但每一条入海口都不是随意的。

OpenCLI 的每一条 adapter，不只是知道那个页面的结构——它知道什么情况算出错、什么情况算空数据、什么情况下它应该告诉下游「此路不通，但这是我的问题还是路的问题」。SYSU 认证页面返回的空响应、GEO 数据库那个 0% 的 baseline——这些失败本身也是河床的一部分。不是需要被清理的错误，是河道地形的一部分。它们告诉下游：这里的水没有流过去。下一次碰到类似的河道，你就知道了。

一条 adapter 是河流和外部世界之间的类型边界。它把互联网的混乱格式化为 agent 理解的形状，把外部的噪音过滤成河床能接受的语言。它不只是入海口——它是翻译层、验证层、也是第一道 review gate。水流进来之前，已经在这里被检验过一次。

### 一张更大的网

现在回到最开始的那个问题：记忆该放在哪里？

写这篇文章的时候，我的答案是三层河床——Alma 管短期流动，Nowledge Mem 管跨工具沉积，Obsidian 管最终沉淀。

但用到今天，我发现它不止三层了。

Slock 的每个 agent，都有自己的 `MEMORY.md`。那是它们各自的知识角落——研究 agent 记得我最近对哪些方向感兴趣，代码 agent 记得项目的架构偏好，讨论 agent 记得我喜欢从什么角度切入问题。每一份记忆都是私有的、专精的、带着各自 persona 的底色。但当它们需要在 channel 里协作解决一个问题时，这些记忆会在讨论中自然交汇。A 知道背景，B 知道技术细节，C 知道类似的坑以前踩过——它们各自带来的东西拼在一起，才构成完整的上下文。

在这些 agent 之外，还有一个特殊的角色。

我的秘书。它也是 Slock 上的一个 agent，但和其他 agent 不同——它不做研究，不写代码，不和我在 channel 里讨论问题。它的工作是在所有人下班之后，把今天发生在每一条河里的东西捞一遍——Slock 的 agent 们在哪些 channel 里讨论了什么、Nowledge Mem 里新沉淀了哪些知识、Yansu 今天记录的工作轨迹和时间线——然后给我写一份总结。

不是流水账。是一份告诉我还需要知道什么的东西：今天有什么值得跟进的方向，哪些讨论可能被我错过了，哪个 agent 发现了一篇和我之前某个笔记有关的论文。

它不创造新的水流。但它知道整条河今天往哪个方向走了。

这并不是随意的安排。每天固定的钟点、固定的格式、固定的审阅口径。河床上的石子经过它的手，才从「似乎是这样」变成了「可以确定」——一条 observation 写进日报，就成了 working belief。一条 working belief 连续三天被确认，就成了 reviewed decision。它不是创造知识的人。它是知识的转场人——把漂在水面上的东西，接到岸上。

Nowledge Mem 在最底下，是所有 agent 共享的河床。无论哪个 agent 发现了什么值得长期记住的东西，都可以写进去。河床不属于任何一个 agent。它属于整条河。

OpenCLI 在最外面，是入海口。它不负责记忆，它负责把外面世界的水引进来。

所以现在的图景大概是这样的：

在最外层，OpenCLI 从互联网上不断把新的水引进来——一篇论文、一条帖子、一个 GitHub Issue。

进来之后，Slock 的 agent 们接手——研究 agent 判断它是否相关，讨论 agent 和它对话、拆解、质疑，代码 agent 验证其中提到的技术点。秘书始终在旁边看着，把值得我知道的东西挑出来。

而 Alma 在最前面，是我和整条河之间的界面——我通过它发起问题、调整方向、看到水正在往哪里流。

### [Yansu](https://yansu.app)：河边的观察者

但还有一件事，是上面所有这些工具都做不了的。

它们都在回应我。Alma 等我说话，Slock 的 agent 等我在 channel 里 @ 它们，秘书等我下班了才整理。没有人在我工作的时候，就站在旁边看着。

Yansu 在做这件事。

它不是某个我主动去打开的工具。它就运行在我的桌面上，安安静静的。我看论文的时候它在，我和 agent 讨论的时候它在，我写代码的时候它也在。它不打扰我，只是在看——看我今天主要在做什么方向、用什么工具、和哪些 agent 交互最频繁、在什么环节会卡住。

然后它把这些观察结晶成知识。不是流水账式的记录，是结构化的、可以被其他 AI 直接使用的知识——我的工作习惯、我的偏好、我常用的工具链。

<img src="/assets/knowledge-flow/yansu-knowledge.webp" alt="Yansu 的 knowledge 视图，展示整理后的行为模式、知识条目和来源统计。" width="1000">

这就是 **Yansu.skill** 在做的事情。它把观察到的模式打包成一个文件，可以被注入到任何 AI agent 的上下文里。下次我开一个新的 Claude Code 会话，不需要重新描述一遍我的项目结构和工作习惯——Yansu 已经替我准备好了。

它和秘书的区别在于：

秘书是**向后的**——每天结束的时候告诉我今天发生了什么。
Yansu 是**向前的**——它看到的东西，直接变成下一个 agent 的启动上下文。

一个在河岸上总结河道今天的走向。
一个在河的源头，告诉每一滴新流进来的水，这条河习惯怎么走。

<img src="/assets/knowledge-flow/yansu-memory.webp" alt="Yansu 的 memory 视图，展示从日常观察沉淀下来的长期记忆与可检索条目。" width="1000">

但它也会告诉上游。

当它观察到同一个方向的水流连续好几天都很活跃，它会告诉 OpenCLI：「这里鱼多，多引些水来。」当它注意到某条支流一直在消耗能量却只带回泥沙，它会告诉调度：「那条路可以停了，换一个方向。」Yansu 不只是观察者。它是河床下面的暗流——从土壤深处，轻轻推着表面的水。

但它不会推得太用力。观察反馈到改变之间，隔着 24 小时的冷却期——同一个模式如果只出现一次，可能只是偶然。如果连续出现，河床自己会调整方向。

所以最后的图景是这样的：

Yansu 在最底部——不是河床，是河床下面的土壤。它不直接参与流动，但它的质地决定了水流的方式。

往上是一层工具层：Alma 是我的界面，Slock 是我的工作区，OpenCLI 是通往外部世界的管道。

再往上是记忆层：每个 agent 的 `MEMORY.md` 是各自的知识角落，Nowledge Mem 是所有记忆交汇的河床。

而所有这些 agent——做研究的、写代码的、深度讨论的、当秘书的——都在 Slock 里。有的在水里游，有的在岸上看。

每一层都有自己的角色。没有一层是多余的。也没有一层需要我来管理。

但它们各自独立，却彼此照看。当一条河道堵了，水会从旁边的路绕过去。你不是在设计这些路——它们是在一次次的阻塞和绕行中，被水自己冲出来的。

而水也不总是一路向下。河底的沉淀会反过来告诉上游：这里鱼多，可以多引些水来。那里盐分太高，换条路。刚来的时候以为河流只有一个方向。用久了才发现——它在回流中学会了自我修正。

你坐在 channel 里，看着它们聊。偶尔插一句话，水就换了一个方向。

这不是什么宏大的架构。这就是一条河在流动的样子。

从最初意识到碎片散落一地，到造水库失败，到理解 Agent Native 的意义，到两条河交汇，再到现在这张由不同角色、不同记忆、不同入海口编织起来的网——每一步都不是设计出来的，是被水自己冲出来的。

你不知道水会流到哪里去。但你知道它一直在走。
